如何更有效地使用 Claude Code

发布于 2026-05-21 Claude Code AI 效率工具 开发工具 

前言

Claude Code 是 Anthropic 推出的命令行 AI 编程助手，它不仅仅是一个代码补全工具，更是一个能够理解项目上下文、自主完成复杂任务的智能搭档。用了几个月之后，我总结了一些让 Claude Code 发挥最大效用的方法。

1. 写好 CLAUDE.md 是效率的基石

CLAUDE.md 是 Claude Code 进入项目后最先读取的文件，相当于给 AI 的一份”项目入职指南”。写得越好，AI 的表现越精准。

我推荐的 CLAUDE.md 结构：

## 常用命令
# 以表格形式列出高频命令，AI 可以快速查找

| 命令 | 说明 |
|------|------|
| npm run dev | 启动开发服务器 |
| npm test | 运行所有测试 |

## 项目架构
# 简述目录结构和技术栈

- src/components/ — UI 组件
- src/services/ — API 服务层

## 注意事项
# 列出容易踩坑的地方

技巧： 把”不要做的事”也写进去，比如”不要直接修改 themes/xxx/_config.yml，用项目根目录的替代配置文件”，能有效避免 AI 自作主张。

2. 需求描述要具体，不要模糊

这是使用 AI 工具的通用原则，但在 Claude Code 中尤其重要。因为它真的会按你说的去做——模糊的需求会导致模糊的结果。

# 不好的描述
> 优化一下这个组件

# 好的描述
> 把 UserList 组件中的搜索功能改为防抖 300ms，
> 并在搜索时显示 loading 状态，用 Suspense 包裹

技巧： 用”文件路径 + 具体行为 + 约束条件”的三段式描述，几乎总能得到符合预期的结果。

3. 善用权限系统，减少打断

刚上手时每次操作都要点确认，确实有点烦。可以渐进式地配置权限：

  • 对于 git status、git diff、npm test 等只读或安全的命令，设为 allow
  • 对于 git push、npm publish、rm -rf 等高风险操作，保持 ask 或设为 deny
  • 对于 git commit，视项目情况决定

配置文件在 .claude/settings.json：

&#123;
  "permissions": &#123;
    "npm test": "allow",
    "npx hexo server": "allow"
  &#125;
&#125;

4. 充分利用斜杠命令

几个经常被忽略但很好用的命令：

命令 场景
/compact 对话太长、上下文快满时，压缩释放 token
/review PR 提交前自检代码质量
/verify 改完代码后，启动项目验证功能是否真的生效
/init 新项目快速生成 CLAUDE.md 骨架
/run 一句话启动项目，自动识别项目类型

技巧： 养成改完代码后跑 /verify 的习惯，AI 会启动应用并在浏览器中确认效果，比自己手动测快得多。

5. 把重复的工作交给 Skill

如果你经常做某种操作——比如每天写日报、每次部署前跑一堆检查——可以创建或安装 Skill。

举个例子，我安装了 daily-report skill 之后，每天下班前只需要：

/daily-report

它就会自动收集当天的 git 提交、文件变更，生成结构化的日报。省去了手动翻 git log 和回忆今天干了什么的麻烦。

你也可以用 skill-creator 来创建自己的专属 Skill。

6. 复杂任务先进入计划模式

面对多文件重构、新增功能模块等复杂任务，不要直接让 AI 开写。先触发计划模式（AI 在不确定时会自动进入，也可以主动说”先不用写代码，给我一个方案”），让它：

  1. 先探索相关代码
  2. 给出实现方案
  3. 你审阅确认
  4. 再开始编码

这样做的好处是：你审批的是方案，不是代码。方案阶段改方向成本极低，代码写完了再改就费劲了。

7. 用记忆系统减少重复交代

遇到过这种情况吗——每次新建会话都要重新解释项目背景？

Claude Code 的记忆系统就是为这件事设计的：

> 记住：我们的 API 使用 JWT 认证，token 有效期 24 小时，
> 刷新 token 的接口是 POST /auth/refresh

之后在任何会话中，AI 都会带上这段上下文。适合存放：

  • 项目技术决策的背景原因
  • 外部系统链接（看板、监控面板等）
  • 团队约定的编码规范

8. Agent 并行处理独立任务

当你有多个互不依赖的任务时，Claude Code 可以并行处理。比如：

> 同时在以下文件中各添加一个导出函数：
> - src/utils/format.ts
> - src/utils/validate.ts
> - src/utils/cache.ts

AI 会自动分配多个 Agent 并行编辑这三个文件，比串行处理快得多。

9. 让 AI 解释而非猜测

遇到不熟悉的代码时，不要直接让 AI 改，先让它解释：

> 解释一下 src/middleware/auth.ts 的认证流程，用中文

理解之后再决定要不要改、怎么改。这样既学了代码，又避免了 AI 在不完全理解业务逻辑的情况下乱改。

10. 版本控制是安全网

Claude Code 虽然强大，但不是每次都能一次做对。我习惯：

  • 改代码前确保工作区是干净的（git status 无未提交文件）
  • 每个独立任务完成后单独 commit
  • 出问题时用 git diff 查看 AI 改了哪些，做到心中有数

这样即使 AI 方向跑偏了，一个 git checkout -- . 就能回到安全状态。

总结

用好 Claude Code 的关键不在于掌握多少命令，而在于：

  1. 写好 CLAUDE.md——这是投入产出比最高的一件事
  2. 学会写清楚需求——把 AI 当同事，而不是读心术大师
  3. 建立自己的工作流——Skill + 斜杠命令 + 记忆系统配合使用
  4. 保持人对代码的控制权——AI 是加速器，不是替代品

用这些方法之后，我的日常开发效率提升了大概 3-5 倍——而且最爽的是，把那些机械重复的事情交给 AI，自己可以专注于真正需要思考的设计和决策。
